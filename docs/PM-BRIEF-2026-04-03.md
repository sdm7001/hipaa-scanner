## Architecture Decision Summary

**Product**: Multi-tenant HIPAA network compliance scanner with MSP portal (cloud) and collector agents (on-premise).

**Key architectural decisions**:

1. **Agent-to-portal communication via HTTP polling** (not WebSocket/gRPC). Agents make outbound HTTPS requests only -- no inbound ports needed on client firewalls. Heartbeat every 5 minutes carries command queue responses (scan-now, update-config).

2. **HMAC-SHA256 agent authentication** with per-agent secrets established during one-time enrollment. Simpler than mTLS, works through corporate proxies. mTLS available as opt-in for high-security environments.

3. **Network discovery via nmap** (python-nmap wrapper). Already in dependencies but never called. Ping sweep + port scan + OS detection determines which connector (WinRM/SSH/SNMP) to use per host.

4. **Three new connectors**: SSH (Paramiko), SNMP v2c/v3 (pysnmp), HTTP (httpx). All pure Python, no native dependencies, package cleanly with PyInstaller.

5. **73 new HIPAA checks** bringing total to 101. Covers MFA, backup, email security, USB policy, VPN, wireless, certificates, SIEM, browser security, AppLocker, database encryption, physical/administrative safeguards, network device hardening, and Linux-specific checks.

6. **Windows agent as pywin32 service** packaged via PyInstaller into single EXE, wrapped in WiX MSI. Linux variant uses systemd. Both share identical Python codebase with OS-conditional imports.

7. **WebSocket for UI scan progress** (browser to FastAPI, same-origin -- no firewall issues). APScheduler for report scheduling (PostgreSQL job store). Audit log as append-only table with 6-year retention.

8. **Existing stack fully preserved**: FastAPI, SQLAlchemy async, PostgreSQL, React/TS/Vite, Tailwind, WeasyPrint PDFs, JWT auth, BaseCheck plugin architecture. No framework changes.

## Phase 2 -- Collector Agent MVP (TODO List)

Assign to a **backend/systems developer** (different from Phase 1 if possible -- these can run in parallel after tasks 1-4 of Phase 1 are done).

1. **Create agent package structure**
   - New directory: `scanner/hipaa_scanner/agent/`
   - New files: `__init__.py`, `core.py`, `enrollment.py`, `config_manager.py`, `service_win.py`, `service_linux.py`, `updater.py`
   - `core.py` is the main loop: initialize -> enroll (if not enrolled) -> start heartbeat thread -> start scan scheduler -> wait for commands.

2. **Implement enrollment protocol**
   - `enrollment.py`: Generate Ed25519 keypair, POST to /api/v1/agents/enroll with one-time token, store returned agent_id + agent_secret in encrypted local config.
   - Backend: `backend/app/api/v1/agents.py` -- POST /api/v1/agents/enroll-token (generates token, stores hash), POST /api/v1/agents/enroll (validates token, stores agent, returns credentials).

3. **Implement HMAC authentication**
   - New file: `backend/app/auth/agent_auth.py` -- FastAPI dependency that validates `X-Scanner-API-Key` (agent_id) + `X-Agent-Signature` (HMAC) + `X-Agent-Timestamp` (replay protection).
   - Agent side: `scanner/hipaa_scanner/agent/core.py` -- sign every request with HMAC-SHA256(agent_secret, body + timestamp).

4. **Implement heartbeat loop**
   - Agent: `core.py` -- HeartbeatThread sends POST /api/v1/agents/{id}/heartbeat every 5 minutes. Parses response for commands.
   - Backend: `agents.py` -- POST /api/v1/agents/{id}/heartbeat updates agent record (last_heartbeat_at, status, metrics), returns pending commands from a simple DB-backed queue.

5. **Implement network discovery**
   - Refactor `scanner/hipaa_scanner/engine.py`: split into discovery phase (nmap ping sweep + port scan + OS detection) and scan phase (check execution).
   - Agent uploads discovery results to new endpoint: POST /api/v1/clients/{id}/discovery (or included in scan upload payload).

6. **Implement scan execution and upload**
   - Agent: receives "scan-now" command via heartbeat -> runs discovery -> runs checks -> uploads results via existing POST /api/v1/scans/upload with HMAC signing added.
   - Backend: modify upload endpoint to accept HMAC auth in addition to API key auth.

7. **Windows service wrapper**
   - `service_win.py`: Use `win32serviceutil.ServiceFramework` to register as Windows service. Handle start/stop/pause.
   - Fallback: if pywin32 not available, use NSSM to wrap the Python process as a service.

8. **PyInstaller packaging**
   - New file: `installer/windows/pyinstaller.spec` -- spec file for single-file EXE build.
   - Test: build on Windows, verify EXE runs, verify service registration works.

|-------|-------------------|-----|
| Phase 1 (tasks 1-8) | Backend Python dev familiar with FastAPI, SQLAlchemy, Alembic | Pure backend fixes and new endpoints. No frontend, no agent. |
| Phase 2 (tasks 1-8) | Systems/DevOps Python dev familiar with Windows services, networking, nmap | Agent architecture, service registration, network scanning. Different skill set from web API development. |
| Phase 3 (new checks) | Can be split across 2-3 devs by batch | Each batch is independent. Batch 1 (WinRM) can start immediately. Batch 2-4 require new connectors first. |

| Phase 5 (commercial) | Full-stack + business logic | Branding, email delivery, billing hooks. |

**Parallelization**: Phase 1 tasks 1-4 (bug fixes) should be done first. After that, Phase 1 tasks 5-8 (new endpoints) and Phase 2 (agent) can run in parallel. Phase 3 Batch 1 (WinRM checks) can start immediately since it uses the existing connector. Phase 4 (frontend) can start as soon as Phase 1 APIs are done.
